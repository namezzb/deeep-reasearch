# Long-Running Agent Harness 设计文档

## 1. 背景与目标

### 问题陈述

Claude Code 在离散会话中工作，每个新会话开始时对之前发生的事毫无记忆。如同软件项目由轮班工程师负责，每位工程师到达时不了解前班发生了什么。由于上下文窗口有限，复杂项目无法在单个窗口内完成。

### 解决方案

采用双管齐下的模式：
1. **初始化智能体（Initializer Agent）**: 首次会话设置环境
2. **编码智能体（Coding Agent）**: 后续会话增量开发

## 2. 核心组件

### 2.1 feature_list.json

功能需求清单，JSON 格式，便于模型理解和修改。

```json
{
  "features": [
    {
      "id": "core-001",
      "name": "功能名称",
      "passes": false,
      "test_steps": ["步骤1", "步骤2"]
    }
  ]
}
```

**规范**:
- 仅修改 `passes` 字段（true/false）
- 禁止删除功能条目
- 使用 JSON 格式避免模型误改

### 2.2 claude-progress.txt

进度追踪文件，记录已完成工作。

```
[2026-04-15] [core-001] 完成项目初始化
[2026-04-15] [core-002] 实现进度追踪系统
```

**规范**:
- 时间戳格式: `[YYYY-MM-DD HH:MM]`
- 功能ID格式: `[FEATURE-ID]`
- 每次会话结束必须更新

### 2.3 init.sh

环境初始化脚本，确保开发环境一致。

```bash
#!/bin/bash
# 检查依赖
# 安装依赖
# 验证环境
# 运行基础测试
```

### 2.4 resume.sh

状态恢复脚本，每次会话开始执行。

```bash
#!/bin/bash
# 读取进度文件
# 读取 git 历史
# 检查功能状态
# 初始化环境
```

## 3. 会话工作流

### 3.1 进入状态步骤

```
1. pwd - 确认工作目录
2. tail -20 claude-progress.txt - 了解近期进展
3. git log --oneline -10 - 了解提交历史
4. cat feature_list.json - 了解功能状态
5. bash init.sh - 初始化环境
```

### 3.2 开发步骤

1. **选择功能**: 从 `feature_list.json` 选择 `passes: false` 的最高优先级功能
2. **制定计划**: 创建实现计划（RED）
3. **编写测试**: 编写功能测试（RED）
4. **实现功能**: 通过测试（GREEN）
5. **重构优化**: 改善代码质量（IMPROVE）
6. **验证**: 确保功能完全通过

### 3.3 结束步骤

```
1. 运行完整测试套件
2. 更新 feature_list.json (passes: true)
3. 更新 claude-progress.txt
4. Git 提交（描述性消息）
5. 验证环境干净（可合并状态）
```

## 4. 四大故障模式与对策

| 故障模式 | 对策 |
|---------|------|
| 智能体过早宣布完成 | 功能列表明确每个功能的 test_steps，必须全部通过才标记 passes: true |
| 环境有 bug 或未记录 | 必须运行测试，提交前验证环境可用 |
| 过早标记功能完成 | 强制自我验证，测试驱动开发 |
| 浪费时间摸索环境 | init.sh 提供完整的环境启动方案 |

## 5. 最佳实践

### 5.1 增量开发

**每次只处理一个功能**，避免上下文溢出。

### 5.2 测试验证

- 使用 TDD 流程
- 功能必须通过实际测试才标记完成
- 避免"看起来可以"的主观判断

### 5.3 Git 纪律

- 使用描述性提交消息
- 每次会话至少一次提交
- 确保提交可回滚

### 5.4 环境清洁

会话结束时确保：
- 无未处理的错误
- 无 major bugs
- 代码可合并到主分支
- 文档已更新
